需求分析：实际痛点

系统要解决的是高校教务数据分散、角色边界不清和过程追踪困难的问题。

数据分散

用户、学院、专业、行政班、课程、教学班、课表、成绩、考勤和预警数据容易散落在不同表格或人工记录中。

角色协作弱

管理员、教师、学生关注的数据不同，需要统一权限边界和同一套基础教务数据。

过程追踪难

成绩、考勤、课表和预警等过程数据如果不结构化，后续统计和追溯成本高。

核心需求可维护、可查询、可统计、可导入导出、可按角色和数据范围控制访问。

痛点。这一页把原来空泛的“场景”和“核心问题”合并。重点是说明为什么需要数据库系统：不是因为要做页面，而是因为业务对象多、数据关系多、角色协作多，必须有统一的数据结构。

需求分析：拟解决方案

解决方案是按管理员、教师、学生划分功能，再把功能落到数据库表和接口。

管理员

负责用户、角色、权限、公告、系统配置和基础教务数据管理。

sys_user college sys_permission announcement

权限落点：账号、角色、权限码、权限菜单路径和系统配置。

教师

负责本人教学班课表、成绩、考勤和学业预警管理。

teaching_class student_profile attendance_record academic_warning academic_warning

权限落点：按教学班和学生名单组织过程数据。

学生

查看本人课程、课表、校历、成绩、绩点、考试、补考和预警。

course class_schedule grade_record schedule

权限落点：按当前登录学生过滤本人查询数据。

解决方案。这里要明确回应你的要求：项目拟解决方案里必须有三角色的数据库设计。讲法是先按角色拆业务，再说明每类角色对应哪些表，最后引到权限控制和演示账号。

数据库总览。这一页只讲数据库设计方法。现在不具体讲某张表，下一页开始用课程表完整展示从业务对象到表结构的过程。

课程表设计流程 I：业务对象到实体关系

课程先作为教学资源实体，再连接教学班、课表和成绩。

业务对象课程：课程编码、名称、学分、学时
course 实体code / name / credit / hours
class_schedule教学班关联课程
grade_record成绩按课程和教学班统计
schedule课表安排课程
为什么单独建 course课程会被教学班、课表和成绩重复引用，如果只写在成绩表中会造成课程名称重复和维护困难。
实体边界课程保存稳定的教学资源信息；教学班、成绩、课表保存围绕课程发生的业务记录。

课程设计一。这页不是直接贴 course 表，而是先讲为什么课程是独立实体。课程会被考试、成绩和课表引用，所以它应该作为基础表存在，而不是把课程名散落在各个业务记录里。

课程表设计流程 II：逻辑表到约束索引

course 表保留课程基础信息，并通过唯一编码、学院关联和逻辑删除维护。

字段（数据类型）设计含义
id（INT UNSIGNED）课程主键，自增生成，用于被教学班、成绩、课表等业务记录引用。
name（VARCHAR(100)）课程名称，例如数据库系统、软件工程，是页面展示和下拉选择的核心字段。
college_id（TINYINT UNSIGNED）课程所属学院和课程性质，支撑课程归类、筛选和统计。
hours（INT UNSIGNED）课程学时，表示课程计划时长，为教学安排和课表编排提供参考。
description（VARCHAR(500)）课程别名或补充说明，保存课程展示名称和扩展信息。
create_time（DATETIME）记录课程创建时间，方便追踪基础数据产生时间。
update_time（DATETIME）记录课程最后修改时间，方便维护和排查数据变更。
deleted（TINYINT UNSIGNED）逻辑删除标记，0 未删除、1 已删除，删除课程时保留历史引用。

课程设计二。这页完整讲 course 表的字段。重点讲 name、college_id、hours 和 deleted 为什么存在。最后说明课程表通过 id 被考试、成绩和课表引用，完成从业务对象到逻辑表的闭环。

后端框架和设计

后端采用 Spring Boot + Spring Security + MyBatis-Plus 的分层结构。

Controller接收 REST 请求，校验权限，统一返回 ApiResponse。
Service封装业务逻辑：数据范围、导入导出、统计、分页、逻辑删除。
Mapper通过 MyBatis-Plus 访问 MySQL 表结构。
Databaseschema.sql、views.sql、data.sql 建库建表、创建视图并提供演示数据。

Spring Boot 3.3.7

后端基础框架与接口组织。

MyBatis-Plus

对象和数据表映射，支持分页与条件查询。

JWT + Security

Token 登录、接口拦截、角色权限和数据范围控制。

Knife4j / POI

接口文档、Excel 模板、导入和导出能力。

后端。这一页讲技术框架和 MVC 三层。不要展开代码，只说明 Controller、Service、Mapper 分别做什么，并点出 JWT、AOP、Knife4j、POI 这些支撑能力。

前端框架和设计

前端采用 React + Axios + TanStack Query 的标准管理端开发流程。

Pages页面表格、筛选表单、弹窗、导入导出按钮，承接具体业务操作。
React Router登录后进入 AppShell，根据后端菜单访问课表、课程、成绩、考勤等页面。
Axios统一封装请求，携带 Token，集中处理响应和错误提示。
REST API调用后端资源接口，最终完成数据库读写。

React 19

组件化组织管理端页面。

AuthProvider

保存用户信息、Token 和登录状态。

Tailwind CSS

配合本地 UI 组件实现表格、表单、弹窗和按钮。

TanStack Query

管理服务端状态，支撑列表查询、刷新和加载状态。

前端。这一页讲前端不是重点宣传页面，而是说明行业常见开发流程：页面触发操作，Axios 请求后端，后端处理数据库，结果回到页面展示。
